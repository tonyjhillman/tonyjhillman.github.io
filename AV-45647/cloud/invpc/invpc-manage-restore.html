Restore a Backup

    +
    A backup can be restored onto the same database that it was created on, or it can be restored onto another database.

    This information is for anyone still using Couchbase Server 6.6, hosted in their own cloud provider’s VPC. It does not apply to Couchbase 7.0 or later, hosted in Couchbase’s VPC and fully managed by Couchbase.

    All of the information for users of Server 6.6 self-managed in their own cloud provider’s VPC, such as this page, can be accessed at this index page.

    Guidelines and Limitations

    • The source database that originally created the backup must still exist and be accessible within the Capella UI.

      Although backups are preserved in the connected cloud even after a database is deleted, the only user-accessible method for restoring a backup requires that you be able to access the source database’s Backup & Restore tab.

      If you need to restore a backup from a database that is no longer available in the Capella UI (such as after an accidental database deletion), please contact Couchbase Capella Support for assistance. In the meantime, it is highly recommended that you download the backup from cloud storage so as to ensure it is not deleted as part of the database’s retention policy.

    • For customers hosting Capella in their own cloud provider’s VPC, a backup can only be restored to a database that is deployed in the same connected cloud as the source database that created the backup.

    • Backups can only be restored to a database that is running the same major version (or higher) than the source database that originally created the backup.

    • Data can only be restored to buckets that have the same names and conflict resolution methods as the buckets from the backup. Ensure that you’ve configured buckets on the target database that have the same names and conflict resolution methods as the ones you are trying to restore data from. If a bucket of the same name does not exist on the target database, no data from that bucket will be restored from the backup.

      Any conflicts that occur during the restore will be resolved automatically using the conflict resolution method configured for the buckets. This means that if a key-value pair in the target bucket is newer than the one in the backup, then the one from the backup is not restored.

      If the goal of the restore is to overwrite the current data with the data from the backup, it is recommended that you restore the data to a brand new bucket of the same name on another database. That way, all data will be restored in the state that it is in the backup. From there you can copy the data over to the desired bucket as you see fit.

    • When restoring from a Series backup, data will be restored from the latest Incremental backup in the series. The restore process will take into account any failovers that may have occurred in between the time that the backups were originally taken. If a failover did occur in between the backups, and the backup archive contains data that no longer exists in the database, then the data that no longer exists will be skipped during the restore. If no failovers occurred in between backups, then restoring a Series backup will restore all data from the series.

    • Restoring database backups is supported during rebalances and failovers. If a rebalance is in progress, the restore process will track any data movement around the database and ensure that data is restored to the appropriate node. If a failover occurs during restore, the restore process will recover so long as the failed node is removed from the database within three minutes. If the failed node is not removed, then the restore will fail.

    Restore Bucket Data

    Permissions Required

    To restore bucket data from a backup, the following permissions are required:

    • You must have the Project Owner role for the project containing the source database that created the backup, as well as the project containing the destination database of the restore.

    1. Go to the Backup & Restore tab of the source database that created the backup that you wish to restore.

      1. Go to the Databases tab in the main navigation.

      2. Find and click on the source database.

        This opens the database with its Overview tab selected.

      3. Click the Backup & Restore tab.

    2. In the Backups list, find and click on the backup that you wish to restore.

      This opens the backup’s fly-out menu.

    3. Click Restore backup.

    4. Specify the Restore Location.

      The restore location is the database that you wish to restore the backup to. This can be the same database that created the backup, or it can be another database in the same connected cloud.

      1. Use the Project dropdown menu to select the project that contains the database that is to be the destination of the restore.

      2. Use the Database dropdown menu to select the database that is to be the destination of the restore. Only databases that are located in the same connected cloud appear in the dropdown menu.

      Once you’ve selected the database, click Next.

    5. To confirm that you’re restoring to the desired database, enter the name of the destination database.

    6. Once you’re ready, click Restore to begin restoring the backup.

      Note that there can be a slight delay while the restore is scheduled, after which the fly-out will close.

    Restoring Indexes

    If GSI indexes were included in the bucket that you restored, they are automatically restored in a round-robin fashion among the current nodes running the Index Service. However, these indexes will only be created, not built.

    The reason that indexes are created and not built is because Couchbase Capella does not know the optimal index topology ahead of time. By not building the indexes, it gives you the option to manually move each index between nodes and build them yourself.

    However, if you find the automatic index distribution acceptable, you can use the Capella UI to rebuild each index.

    Permissions Required

    In order to access indexes in the Couchbase Capella UI, the following roles are required:

    • You must have the Project Owner or Database Data Reader/Writer role for the project that contains the database.
    To Rebuild an Automatically-Restored GSI Index

    1. Go to the database’s Tools > Indexes tab.

      1. Go to the Databases tab in the main navigation.

      2. Find and click on the database on which you restored a backup.

        This opens the database with its Overview tab selected.

      3. Click the Tools > Indexes tab.

    2. Each index that was restored will display a status of Created. Click the Play icon in the Status column for each index that you wish to rebuild.
